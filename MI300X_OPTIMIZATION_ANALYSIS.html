Summary

Well-Optimized GOOD

  • Core MFMA dispatch (16x16x16, 16x16x32, 16x16x128 for FP8)
  • Wave64 mode for CDNA
  • Cache management (acquire/release mem)
  • Multi-XCC support
  • FP8 conversion intrinsics
  • AccVGPR handling
  • ELF kernel descriptor packing

Major Gaps PERFORMANCE IMPACT

  1. No double-buffered global→LDS pipeline in auto-generated kernels
  2. No persistent kernels / Stream-K work distribution
  3. No 32x32 MFMA tile variants in tensor core definitions
  4. No LDS bank conflict avoidance (swizzle) in auto-generated code
  5. No INT8 tensor core path
  6. No fine-grained s_waitcnt emission in codegen
  7. Bandwidth utilization very low (~128 GB/s vs 5.3 TB/s peak) for general kernels
  8. No structured sparsity (2:4 sparsity) support despite hardware support
  9. LDS bank conflict swizzle not used (ds_swizzle_b32)

The Hand-Tuned Benchmarks Prove These Are Solvable

The extra/gemm/cdna_asm_gemm.py (1000+ lines of hand-written ISA) demonstrates:

  • 256x256 tiles with 256 workgroups
  • Double-buffered PGR pipeline
  • Stream-K partitioning
  • LDS bank conflict avoidance via manual register placement
  • Stagger-unroll for L2 cache hit rate
  • s_setprio(3) for maximum priority

The gap is between what the general auto-generated path achieves and what hand-tuned ISA demonstrates is possible on the hardware.
